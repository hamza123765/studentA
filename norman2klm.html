Comparative Framework: Norman · GOMS · KLM

Three complementary models for interaction analysis: cognitive diagnosis (Norman), task structure (GOMS), and execution-time prediction (KLM).

Table 1: Model-based evaluation – what each model explains vs. predicts
Model Category What it Explains / Diagnoses What it Predicts / Measures
Norman's Action Cycle Cognitive · Descriptive Gulfs of execution/evaluation, user errors, feedback gaps, seven-stage interaction cycle ❌ No quantitative prediction; identifies usability problems
GOMS Predictive · Analytic Task decomposition, method selection, operator sequences for expert users ✔ Relative efficiency, task structure, procedure comparison
KLM Predictive · Quantitative Low-level motor and mental actions at execution phase ✔ Precise execution time (seconds) using operator constants

KLM Operators: Execution-Level Primitives

The Keystroke-Level Model (KLM) decomposes the execution phase of interaction into a small set of primitive operators. KLM assumes the user is expert, error-free, and that the method has already been chosen. No planning, interpretation, or evaluation stages are modeled.

Table 2: KLM operators with descriptions, time estimates, and examples (Dix et al. Ch.12)
Operator Name Description Time (seconds) Example
K Keystroke Press a key on keyboard 0.20 (skilled typist)
0.28 (average)
1.20 (novice)		 Typing a character, Enter, Delete
B Button press Press or release mouse button 0.10 Left-click, right-click
P Point Move pointer to a target 1.10 Move cursor to icon, menu item
H Home Switch hand between devices 0.40 Keyboard → mouse, mouse → keyboard
D Draw Draw lines using pointing device Variable (depends on path length) Dragging to draw a shape, selection box
M Mental preparation Cognitive preparation for physical action 1.35 Pause to locate control, decide next action
R Response System response time (user waits) Variable (system-dependent) Page load, calculation, file save

KLM prediction formula: Texecute = Σ(K + B + P + H + D + M + R) with appropriate time constants.

Worked Example: Correcting a Typing Error

Task Description

User notices a single-character error in a text document, deletes it, types the correct character, and returns to the insertion point.

Task Decomposition

  1. Move hand from keyboard to mouse
  2. Point mouse cursor after the incorrect character
  3. Click to set insertion point
  4. Return hand to keyboard
  5. Press Delete key
  6. Type the correct character
  7. Move hand back to mouse
  8. Point to new insertion location
  9. Click to reposition cursor

KLM Operator Sequence

H[mouse]   0.40s  – Home: keyboard → mouse
P          1.10s  – Point to error location
B          0.10s  – Button press (click)
H[keyboard]0.40s  – Home: mouse → keyboard
M          1.35s  – Mental: prepare to delete
K[Delete]  0.20s  – Keystroke: delete character
K[char]    0.20s  – Keystroke: type correction
H[mouse]   0.40s  – Home: keyboard → mouse
M          1.35s  – Mental: prepare to reposition
P          1.10s  – Point to new location
B          0.10s  – Button press (click)

Predicted Execution Time

Total = 6.70 seconds (expert, error-free, no system response time).

Note: This models only execution. Planning the correction, interpreting feedback, and evaluating success are not included in KLM.

Why Only Seven Operators?

Scope Limitation: Execution, Not Planning

KLM intentionally omits: goal formation, method selection, perception, interpretation, evaluation, learning, error recovery, and high-level problem solving. These are cognitive activities that occur before and after execution.

Assumptions

  • User is expert (no learning curve)
  • Task is routine (method already known)
  • Error-free performance (no corrections, no mistakes)
  • No perceptual or cognitive delays beyond [M] operator

Relationship to GOMS

KLM is a specialization of GOMS that focuses only on the Operator component. GOMS includes Goals, Methods, and Selection Rules; KLM assumes these are already resolved and models only the execution sequence with precise time values.

Key Takeaways

1. KLM predicts execution time only. It consists of seven primitive operators (K, B, P, H, D, M, R) with empirically derived time constants. It does not model planning, interpretation, or evaluation.

2. Norman explains what, GOMS predicts how, KLM measures how long. Norman diagnoses usability gulfs; GOMS analyzes task structure and method selection; KLM provides quantitative execution-time predictions.

3. KLM is a low-level instance of GOMS. Both assume expert, error-free performance. GOMS operates at the task level; KLM operates at the motor-action level.
